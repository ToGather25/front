# Conventional Commit

변경 사항을 분석하고 컨벤션에 맞게 커밋합니다.

## 커밋 타입 규칙

| 타입 | 사용 시점 |
|------|---------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 (README, CLAUDE.md 등) |
| `style` | 코드 동작 변경 없는 포맷 수정 (CSS, 공백, 세미콜론 등) |
| `refactor` | 기능 변경 없는 코드 구조 개선 |
| `test` | 테스트 추가/수정 |
| `chore` | 빌드, 패키지 설정, 기타 잡무 |

## 실행 절차

### 1단계 — 변경 사항 파악

다음을 병렬로 실행한다:
- `git status` — 변경된 파일 목록
- `git diff HEAD` — 변경 내용 전체
- `git log --oneline -5` — 최근 커밋 스타일 참고

### 2단계 — 파일별 타입 분류

각 변경 파일을 아래 기준으로 타입을 분류한다:

**feat**: 새 컴포넌트/페이지/기능 추가, 새 API 연동
**fix**: 버그 수정, 잘못된 동작 교정
**style**: CSS/Tailwind 클래스 변경, 토큰 클래스로 교체, 포맷 정리 (로직 변경 없음)
**refactor**: 컴포넌트 분리, 로직 재구성, 파일 이동 (기능 변경 없음)
**docs**: `.md` 파일, 주석, CLAUDE.md, 설정 파일 설명
**chore**: `package.json`, `vite.config`, `.env`, config 파일, build 설정
**test**: 테스트 파일

### 3단계 — 그루핑 및 메시지 작성

- **같은 타입의 변경은 하나의 커밋으로 묶는다**
- 타입이 다르면 별도 커밋으로 분리한다
- 여러 파일을 묶을 때 설명은 변경의 공통 목적을 한 줄로 요약한다

**메시지 형식:**
```
<타입>: <설명>
```

**규칙:**
- 설명은 한국어로 작성 (이 프로젝트 언어)
- 명령형 동사로 시작 (추가, 수정, 개선, 변경, 제거 등)
- 50자 이내 권장, 100자 초과 금지
- 마침표 없음

**예시:**
```
feat: 성경 읽기 랭킹/내 구절/내 현황 뷰 추가
fix: 성경 필사 타이핑 오버레이 한국어 입력 수정
style: 전체 페이지 typography 토큰 클래스로 통일
chore: commitlint 설정 파일 추가
```

### 4단계 — commitlint 검증

각 커밋 메시지를 검증한다:
```bash
echo "<커밋 메시지>" | npx commitlint
```

오류가 있으면 메시지를 수정하고 재검증한다.

### 5단계 — 스테이징 및 커밋

타입별로 그룹화된 파일만 선택적으로 스테이징한다:
```bash
git add <파일1> <파일2> ...
git commit -m "<타입>: <설명>"
```

커밋이 여러 개라면 순서대로 반복한다.

### 6단계 — 결과 요약

커밋 완료 후 `git log --oneline -<커밋 수>`로 결과를 확인하고 출력한다.

## 주의사항

- `git add -A` 또는 `git add .` 사용 금지 — 파일을 명시적으로 지정
- `.env`, 시크릿 파일은 절대 커밋하지 않음
- 사용자가 명시적으로 요청하지 않는 한 push하지 않음
- `--no-verify` 플래그 사용 금지
